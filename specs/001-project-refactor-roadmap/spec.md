## User Scenarios & Testing *(mandatory)*

### User Story 1 - 前端代码可维护性提升（Priority: P1）

作为一名开发者，当前 App.tsx 单文件约 2900 行，状态管理、业务逻辑与 UI 渲染全部耦合。我需要将其拆分为职责清晰的 hooks 和纯 JSX 骨架，使每个模块独立可读、可测。

**Why this priority**: 这是后续测试基础设施和安全改造的先决条件。不完成分层重构，核心业务逻辑（AI 调用、数据同步）无法被单元测试覆盖，部署安全问题也难以安全重构。

**Independent Test**: 可通过运行类型检查命令验证重构零破坏——命令通过且所有原有功能在浏览器中正常运行，即证明本故事独立交付价值。

**Acceptance Scenarios**:

1. **Given** 项目处于重构前状态，**When** 完成 hooks 拆分，**Then** `useProjects`、`useAI`、`useDesign` 三个独立 hook 文件存在，各自职责不重叠
2. **Given** hooks 已拆分，**When** 运行类型检查，**Then** 零类型错误，编译通过
3. **Given** 重构完成，**When** 手工回归测试所有功能点（生成幻灯片、导出 PPTX、云端同步），**Then** 所有功能行为与重构前完全一致
4. **Given** App.tsx 瘦身后，**When** 查看该文件，**Then** 文件仅包含 JSX 骨架，不包含业务逻辑函数

---

### User Story 2 - 核心业务逻辑拥有自动化测试覆盖（Priority: P2）

作为一名开发者，我需要为项目中可测试的模块（工具函数、本地存储、PPTX 导出、AI 调用 hook、服务端 API）建立自动化测试，使后续功能迭代和安全改造有信心保障。

**Why this priority**: 阶段三的安全改造（API Key 后端代理）属于高风险改动，没有测试覆盖则任何重构都是在盲飞。测试基础设施是工程质量的底线。

**Independent Test**: 可通过执行测试运行命令验证——所有测试通过且覆盖率达标，即证明本故事独立交付价值。

**Acceptance Scenarios**:

1. **Given** 测试框架已配置，**When** 针对 `dbService` 执行测试，**Then** IndexedDB 的增删改查逻辑全部被用例覆盖并通过
2. **Given** 测试框架已配置，**When** 针对 `pptxService` 执行测试，**Then** PPTX 导出的数据映射逻辑被用例覆盖并通过
3. **Given** hooks 已拆分，**When** 针对 `useProjects` 执行测试（mock 存储层），**Then** 项目 CRUD 和云端同步逻辑被用例覆盖并通过
4. **Given** hooks 已拆分，**When** 针对 `useAI` 执行测试（mock AI 服务），**Then** 大纲生成和幻灯片生成流程被用例覆盖并通过
5. **Given** 测试环境配置完成，**When** 针对服务端 API 执行集成测试，**Then** GET/POST `/api/user-data` 接口行为符合规格并通过
6. **Given** 新测试用例编写完毕，**When** 执行全量测试，**Then** 所有用例通过，无失败项

---

### User Story 3 - 应用具备生产环境部署就绪能力（Priority: P3）

作为一名运维/部署人员，我需要该应用消除已知安全风险（API Key 前端暴露）、支持环境变量配置、具备 CORS 和限流保护，并提供完整的部署文档，使其可以安全地部署到公网。

**Why this priority**: 当前存在 API Key 暴露在前端构建产物中的高危问题，但此改造依赖前两个阶段完成后才能安全执行，因此列为第三优先级。

**Independent Test**: 可通过以下方式验证：构建产物中不含 API Key 字符串；使用非默认端口启动服务器成功；发送超限请求时被限流；部署文档覆盖标准生产环境搭建步骤。

**Acceptance Scenarios**:

1. **Given** 安全改造完成，**When** 检查前端构建产物（dist/），**Then** 产物中不存在任何 Gemini API Key 字符串
2. **Given** 安全改造完成，**When** 用户使用应用调用 AI 功能，**Then** 请求经由后端代理转发，用户浏览器无法直接看到服务端 API Key
3. **Given** 服务器支持端口配置，**When** 设置 PORT 环境变量为非 3000 值并启动，**Then** 服务器在指定端口成功监听
4. **Given** 限流配置完成，**When** 同一客户端在短时间内发送大量请求，**Then** 超出限制的请求收到 429 响应，正常请求不受影响
5. **Given** CORS 配置完成，**When** 来自非允许域的请求到达 API，**Then** 请求被拒绝，允许域的请求正常响应
6. **Given** 生产启动脚本存在，**When** 运行对应脚本，**Then** 服务以生产模式启动，提供静态文件服务和 API 服务

---

### Edge Cases

- 重构过程中若某个 hook 的依赖关系出现循环引用，如何识别并拆解？
- 测试执行时若 IndexedDB fake 实现与浏览器行为不一致，如何处理测试结果的置信度？
- API Key 迁移到后端后，已有用户的本地 IndexedDB 和云端数据（以 API Key hash 为主键）如何平滑迁移而不丢失数据？
- 限流策略应针对 IP 还是用户身份（API Key hash）？两者的误伤范围不同。
- 生产模式下服务启动失败（如端口占用、数据库权限问题）时，错误信息是否足够清晰以便快速定位？

## Requirements *(mandatory)*

### Functional Requirements

**阶段一：前端分层重构**

- **FR-001**: 系统必须将 `useProjects` hook 独立为单独模块，包含用户数据加载、云端同步、项目增删改查全部职责
- **FR-002**: 系统必须将 `useAI` hook 独立为单独模块，包含大纲生成和幻灯片生成全部职责
- **FR-003**: 系统必须将 `useDesign` hook 独立为单独模块，包含设计相关状态管理职责
- **FR-004**: `App.tsx` 完成瘦身后，文件内容必须仅为 JSX 骨架，不包含任何业务逻辑函数定义
- **FR-005**: 重构完成后类型检查必须零错误通过
- **FR-006**: 重构前必须更新 `src/AGENTS.md`，明确定义 hooks 边界和接口契约

**阶段二：测试基础设施**

- **FR-007**: 系统必须引入测试框架，并配置支持浏览器环境模拟和 IndexedDB 模拟的测试环境
- **FR-008**: `dbService` 模块必须具备覆盖所有 IndexedDB 操作的单元测试
- **FR-009**: `pptxService` 模块必须具备覆盖数据转换逻辑的单元测试
- **FR-010**: `useProjects` hook 必须具备使用存储层 mock 的单元测试
- **FR-011**: `useAI` hook 必须具备使用 AI 服务 mock 的单元测试
- **FR-012**: 服务端 `/api/user-data` GET 和 POST 接口必须具备使用内存数据库的集成测试

**阶段三：部署就绪改造**

- **FR-013**: Gemini API Key 必须仅由后端持有，前端构建产物中不得包含任何 API Key 信息
- **FR-014**: 前端配置文件必须移除对 API Key 的构建注入逻辑
- **FR-015**: 服务器必须支持通过环境变量配置监听端口，默认值为 3000
- **FR-016**: `package.json` 必须包含生产启动脚本
- **FR-017**: 服务器必须配置 CORS，明确声明允许的请求来源
- **FR-018**: 服务器必须对 API 端点实施请求频率限制，防止滥用
- **FR-019**: 项目必须提供生产部署文档，涵盖反向代理、进程管理和 HTTPS 配置

### Key Entities

- **Hook 模块**：独立的业务逻辑单元，对外暴露明确的状态和操作接口，不包含 UI 渲染逻辑
- **测试用例**：针对单一功能点的可重复执行验证脚本，分为单元测试（mock 依赖）和集成测试（真实依赖）
- **API Key 代理**：后端持有的凭证管理机制，前端通过中间层调用 AI 服务而不直接接触原始 Key
- **部署配置**：环境变量、CORS 策略、限流规则、启动脚本的集合

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 重构完成后，`App.tsx` 文件行数减少至原来的 30% 以下（目标 900 行以内）
- **SC-002**: 重构完成后，类型检查命令零错误通过，人工回归测试 100% 功能点正常
- **SC-003**: 测试基础设施建立后，核心业务逻辑（数据存储、PPTX 导出、项目管理、AI 调用、服务端 API）全部拥有自动化测试覆盖
- **SC-004**: 测试执行时，所有已编写的测试用例 100% 通过
- **SC-005**: 安全改造完成后，对前端构建产物执行全文搜索，找不到任何 API Key 相关字符串
- **SC-006**: 使用非 3000 端口的 PORT 环境变量启动服务，服务在 5 秒内成功监听指定端口
- **SC-007**: 限流生效后，同一来源连续发送超限请求时，超限请求在 1 秒内收到限流响应，不影响正常频率的请求
- **SC-008**: 部署文档完成后，按照文档从零搭建生产环境，整个过程不需要查阅代码即可完成

## Assumptions

- 三个阶段严格按顺序执行，阶段一未完成不开始阶段二，阶段二未完成不开始阶段三
- 重构阶段（阶段一）原则为"只搬移代码，不改行为"，不在重构过程中引入新功能或修复已知 bug
- 测试阶段（阶段二）覆盖范围以 TODO.md 中明确列出的六个模块为准，不要求 100% 代码行覆盖率，但要求关键业务路径全部覆盖
- API Key 迁移（阶段三）后，用户首次访问时需重新在前端输入 Key（用于身份标识和本地缓存 key），但实际 AI 调用由后端代理完成
- 部署文档以 Linux + Nginx + PM2 为标准环境，其他环境由使用者自行适配
- 现有用户数据（IndexedDB + SQLite）在全部改造完成后仍可正常访问，不存在数据丢失
- 每个阶段完成后，通过在 TODO.md 中标记对应条目为 `[x]` 来记录进度
